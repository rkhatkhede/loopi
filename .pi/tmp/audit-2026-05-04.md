# loopi Codebase Audit

**Date:** 2026-05-04  
**Auditor:** pi-agent  
**Scope:** Full source tree (src/, tests/, agents/, config files)

---

## 🔴 CRITICAL — Bugs & Incorrect Behaviour

### C1. `readDiffFile` dead-code branch (pr.ts:85–88)
**File:** `src/actions/pr.ts`  
**Issue:** The third ternary branch is a duplicate of the first:
```ts
const filepath = existsSync(resolve(_approvedDir, filename))  // 1st check
  ? resolve(_approvedDir, filename)
  : existsSync(resolve(_pendingDir, filename))                  // 2nd check
    ? resolve(_pendingDir, filename)
    : existsSync(resolve(_approvedDir, filename))               // DUPLICATE of 1st
      ? resolve(_approvedDir, filename)
      : null;
```
The third `_approvedDir` check will **never be reached** — if it evaluates to `true`, the first check already caught it. This means files are never looked up in `_approvedDir` after failing `_pendingDir`. **Fix:** change the third check to check `_approvedDir` from a resolved path again, or better, merge the logic.

### C2. Opportunity category enum mismatch (types vs agent spec)
**File:** `src/types/index.ts:52` vs `agents/opportunity-agent.md:32`  
**Issue:** The Zod schema defines:
```
"feature" | "revenue" | "growth" | "tech-debt" | "security" | "performance" | "quality" | "docs" | "architecture"
```
But the opportunity-agent.md instructs the LLM to output:
```
"feature | refactor | fix | test | optimize | docs"
```
**These are completely different enums.** Every opportunity suggested by the agent will fail Zod validation → pipeline error on step 2. **Fix:** align one to the other.

### C3. `typescript` in `dependencies` instead of `devDependencies` (package.json:30)
**File:** `package.json`  
**Issue:** `typescript` is a runtime dependency. End users installing `@loopi-cli/loopi` via pnpx will download TypeScript (~45MB) unnecessarily. The package only needs TS for `build` / `lint` scripts, plus `tsc` for the `prepublishOnly` hook. **Fix:** move to `devDependencies`.

### C4. `statSync` imported but never used (2 files)
**Files:** `src/actions/status.ts:1`, `src/tui/dashboard.ts:17`  
**Issue:** Both files import `statSync` from `fs` but never call it. Dead import, but also a red flag that some intended stat-checking logic was started and abandoned.

### C5. Config keys `git.commitPrefix` and `git.branchPrefix` defined but never consumed
**File:** `src/types/index.ts:226–227, 274–275`  
**Issue:** The config schema defines `git.commitPrefix` (`"feat(agent):"`) and `git.branchPrefix` (`"agent-improvement/"`), but the actual `applyPatch` function in `pipeline.ts` hardcodes:
- Branch prefix: `loopi/` (line 193)
- Commit message: `feat(loopi): ${summary}` (line 206)
**The config values are silently ignored.** This is misleading for users who customise their config.

## 🟠 HIGH — Design & Robustness Issues

### H1. No error boundary around agent subagent calls
**Issue:** If any agent (vision, opportunity, scout, etc.) fails or returns unparseable JSON, the pipeline has no retry/fallback mechanism. The `parseAgentOutput` function throws on failure, which will crash the entire pi agent workflow. A transient LLM error → full pipeline restart.

### H2. `getGit()` cache never invalidated
**File:** `src/actions/git.ts:14`  
**Issue:** The singleton `_git` instance is cached forever. If `getGit(cwd)` is called with a different `cwd`, the old instance is returned. The `resetGit()` function exists but is never called anywhere.

### H3. No pipeline concurrency lock
**Issue:** There is nothing preventing two `loopi run` or watch cycles from executing simultaneously on the same repo. This could cause:
- Simultaneous `git checkout` on same working tree → `fatal: Unable to create '.../index.lock'`
- Conflicting feature branch names
- Opportunity history corruption from concurrent writes

### H4. `workers/patch-generator.ts` deleted but no migration note
**File:** `src/workers/patch-generator.ts` (deleted)  
**Issue:** This file is referenced in README.md's project structure diagram:
```
src/workers/ → Mechanical helpers (patch generation)
```
But was deleted from the working tree. There's a `D` in git status for it. Was this intentional? If so, the README needs updating. If not, the file might have been accidentally deleted.

### H5. `git pull` in `applyPatch` has no conflict handling
**File:** `src/pipeline.ts:190`  
**Issue:** `git.pull("origin", baseBranch)` is wrapped in try/catch but only catches remote errors. If there are **merge conflicts** during pull, the exception goes unhandled and could leave the repo in a conflicted state.

## 🟡 MEDIUM — Test Coverage & Code Quality

### M1. No tests for critical git workflow functions
**Missing tests for:**
- `applyPatch()` — the core dev-branch workflow
- `approvePending()` — end-to-end approve flow
- `rejectPending()` — file deletion
- `promoteToMain()` — dev→main merge
- All functions in `src/actions/git.ts`

Test coverage is ~30% by file count but covers only the parser, CRUD on vision/opportunity, and PR directory operations. The **most critical path** (apply → approve → promote) has zero test coverage.

### M2. `simple-git` import hack
**File:** `src/actions/git.ts:3–4`  
```ts
const simpleGit: SimpleGitFactory = (simpleGitPkg as unknown as SimpleGitFactory);
```
This double-cast is fragile. It works around ESM/CJS interop issues but should be documented with a comment explaining *why* it's needed, and ideally replaced with a clean ESM import once `simple-git` ships ESM.

### M3. Dashboard `onData` key handler redundancy
**File:** `src/tui/dashboard.ts:327–330`  
```ts
if (char === "r" && data === "r") { ... }
```
`char` is `data.toLowerCase()` so `char === "r"` and `data === "r"` are equivalent for lowercase 'r'. This condition is always true when either is true. Meant to distinguish 'r' from 'R'? But `'r'.toLowerCase() === 'r'` is trivially true.

### M4. TUI terminal-size edge case: negative log panel height
**File:** `src/tui/dashboard.ts:80`  
```ts
const logH = bodyH - panelH;
```
If `bodyH < panelH` (very small terminals), `logH` becomes negative, causing `logLineCount = Math.max(logH - 2, 3)` to produce 3 but the rendering loop will misbehave.

## 🔵 LOW — Nits & Polish

### L1. Help text shows placeholder GitHub URL
**File:** `src/cli.ts:121`  
```ts
console.log(`  Learn more: https://github.com/your-org/loopi`);
```
Should point to the actual org.

### L2. `pnpm-lock.yaml` tracked in git
Currently tracked. For a published CLI tool consumed via `pnpx`, the lockfile has no effect on consumers but adds churn to every PR/commit. Consider adding to `.gitignore`.

### L3. `isMain` check in dashboard.ts won't work when compiled
**File:** `src/tui/dashboard.ts:397`  
```ts
const isMain = process.argv[1]?.endsWith("dashboard.ts") || process.argv[1]?.endsWith("dashboard.js");
```
After `tsc` compilation, the source is `.js` and the cwd/path may not end in `dashboard.js`. This guard is unreliable.

### L4. Config test has extra fields not in schema
**File:** `tests/unit.test.ts:53`  
```ts
humanGate: { enabled: true, requireApproval: true, notificationMethod: "contact_supervisor" },
```
`requireApproval` and `notificationMethod` are not defined in `ConfigSchema`. Zod will silently strip them (no error), but it makes the test misleading — it doesn't actually test that those fields are accepted.

### L5. No `engines` field in package.json
```json
{
  "name": "@loopi-cli/loopi",
  "engines": { "node": ">=18" }  // MISSING
}
```
The code uses `ES2022` target features and ESM. Should declare minimum Node version.

### L6. `rejectPending` uses `unlinkSync` — no async alternative
Minor, but for large project deployments, synchronous filesystem calls block the event loop.
